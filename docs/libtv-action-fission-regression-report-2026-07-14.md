# LibTV 动作裂变非重启回归测试

日期：2026-07-14

## 结论

当前按 `canvasId:nodeId` 隔离的并发池主体是健康的：并行上限、跨节点/跨画布隔离、排队取消、运行中取消、失败行重跑和旧任务终态保护均通过。

初次执行 29 个临时主进程集成场景，21 个通过，8 个期望断言失败。8 个失败归并为 4 个生产问题：

1. **高风险：旧 Renderer 快照会擦除 LibTV 动作行的运行锚点或已完成结果。**
2. **高风险：远端 `queryNode` 一次瞬时失败会把已提交甚至已返回结果的任务直接判为失败。**
3. **中风险：准备阶段无法及时取消，会阻塞并发槽位并可能在停止后继续创建远端节点。**
4. **低风险：启动重试文案在下一次运行或停止后残留。**

四项问题均已在后续处理中修复并加入正式回归测试。

远端连接失败遗留节点和本地结果瞬时处理失败也已做适度处理。按产品需求，结果处理中停止后的本地资源回收不再扩展。

## 测试边界

- 使用真实的 `libtv-generation-runner.cjs`、`libtv-generation-task-store.cjs` 和 `canvas-store.cjs`。
- LibTV CLI 与资源下载使用可控适配器，模拟成功、延迟、失败、繁忙、取消和网络抖动，不产生远端任务或费用。
- 覆盖切换画布等价的 Renderer 停止轮询、旧快照保存、返回后读取，以及删除节点/画布。
- 软件进程退出和重启恢复不在本轮范围内。
- 未运行 Playwright：仓库当前没有 `tests/electron` 测试目录，本轮问题均位于主进程状态机和持久化协调层，确定性集成测试比临时搭建 Electron E2E 更直接。
- 临时诊断测试已删除；已确认的问题使用正式测试固化。

## 问题分析

### 1. 旧快照覆盖主进程任务状态

**状态：已修复。影响：高。** 这最接近“切换画布后任务视觉丢失或结果丢失”的用户操作。

LibTV 动作行协调逻辑在 `electron/main/modules/libtv-generation-runner.cjs:758` 只用 Renderer 快照中的 `row.libtvTaskId` 调用 `getTask`。旧快照没有这个字段时，它不会使用已有的 `latestTaskForTarget(canvasId, rowTarget)`，因此：

- 上传或生成期间切换画布触发旧快照保存，可擦除主进程刚写入的任务锚点。
- 主进程已完成并写入结果后，旧快照晚到可再次擦除 `resultUrl`。
- 主进程内存任务仍会继续，但重新进入画布时可能没有任务 ID 可用于视觉恢复；若终态又被旧保存覆盖，结果会真实丢失。

LibTV 现已与第三方 API 的协调规则一致，按 target 查询最新任务并合并活动锚点或终态结果，同时补齐结果宽高。

### 2. 查询异常错误地否定有效远端任务

**状态：已修复。影响：高。** 会把远端仍在运行或已经成功的任务显示为失败，用户重试后可能重复生成并产生额外费用。

`electron/main/modules/libtv-generation-runner.cjs:503` 在每次 `runNode` 后无条件调用 `queryNode`，然后才判断 `runNode` 是否已经带回图片。因此即使 `runNode` 已返回有效 URL，查询接口的一次网络抖动仍会让整个任务失败。已经进入远端运行后的轮询同样没有短暂查询重试边界。

现在优先使用 `runNode` 已得到的有效结果；只有缺少结果时才查询。必要查询使用最多三次、可取消的短退避，不会因为查询重试重新提交生成。

### 3. 准备阶段停止不完整

**状态：已修复。影响：中。** 用户看到任务已停止，但任务池槽位仍被占用，且远端可能出现孤立节点。

`ensureNamedWorkspace` 和项目准备调用没有接收 `AbortSignal`。测试中停止首个任务后：

- 第二个排队任务必须等首个阻塞调用返回后才能开始。
- 首个任务在已经标记停止后仍继续执行了 `createImageNode`。
- 后续信号检查阻止了真正运行，但新建远端节点没有被清理。

Runner 现在只中止当前任务对共享 ensure Promise 的等待，不会取消其他任务共用的工作区/当天画布初始化。节点上传、创建、连线、查询和运行等独占 CLI 调用继续传递任务信号；每个准备步骤后也会检查停止状态。

### 4. 重试状态残留

**状态：已修复。影响：低。** 任务本身继续正确运行，但 UI 会持续显示“正在尝试其他任务，重试中”。

初次进入运行时在 `electron/main/modules/libtv-generation-runner.cjs:492` 写入 `libtv.generating`，退避时在 `:521` 写入重试状态，下一轮 attempt 开始时没有恢复生成状态。停止任务时 `electron/main/modules/libtv-generation-task-store.cjs:111` 也没有清除 `messageCode/messageParams`。

每次新 attempt 现在恢复 `libtv.generating`，停止任务也会清除 `messageCode/messageParams`。

## 资源清理策略

1. 尚未调用 `runNode` 前发生上传、创建或连线失败，会清理本次创建的生成节点和参考节点。
2. 一旦调用过 `runNode`，无论结果是否明确，都会保守保留远端节点，避免误删已受理任务。
3. 本地结果处理瞬时失败会复用同一个远端 URL 最多重试三次，不重新运行远端生成。

## 后续处理结果

### 准备阶段取消

工作区与当天画布的 ensure 操作使用跨任务共享 Promise。不能直接使用单个任务的 `AbortSignal` 终止共享 CLI，否则一个任务停止会误伤其他同时等待的节点。

实现分为两层：

1. Runner 使用可中止的 Promise 等待包装器；当前任务停止后立即退出并释放池槽位，但共享 ensure 继续完成。
2. 节点创建、上传、连线、运行和查询属于任务独占调用，应把 `AbortSignal` 一路传到 `runProcess`，真正终止对应 CLI。
3. 每个共享调用返回后再次检查 signal，禁止停止后的任务继续创建远端节点。

### 远端节点清理

已增加 `runAttempted` 阶段标记。只有尚未调用过 `runNode` 时，上传、建节点或连线失败才自动删除本次创建的生成节点和参考节点。一旦尝试运行，远端是否已受理存在不确定性，因此保留节点。

### 本地结果处理失败

当前采用轻量策略：取得远端结果 URL 后，本地下载与缩略图失败会在同一任务内最多重试三次，复用相同 URL，测试确认只调用一次 `runNode`。如果连续三次仍失败，任务才进入失败状态。

### 结果处理中停止

按当前产品需求不再处理该极低概率操作，也不为此增加 Asset Store 删除接口和额外资源生命周期复杂度。
